# 血糖监测与健康管理系统软件设计

## 一、项目概述

本项目是一套面向日常血糖管理需求的移动端健康管理应用，作为基于 STM32 的无创血糖检测硬件的配套软件终端。硬件端通过传感器采集人体生理信号，经 STM32 计算得出血糖数值后，通过 HC-05 蓝牙模块将数据传输至手机端。移动端负责接收、存储、可视化血糖数据，并结合用户的身体档案与运动数据，在本地完成热量消耗估算和健康状况评估，向用户提供个性化的运动与饮食建议。

整个软件的数据处理和决策判定均在本地离线完成，不依赖任何云端服务，从设计上保证了用户健康数据的隐私安全。

## 二、技术栈

本项目移动端采用以下技术构建：

- 开发框架：Flutter（Dart 语言），支持 Android 平台原生运行
- 状态管理：flutter_riverpod，基于 Provider 模式实现响应式数据驱动
- 本地数据库：sqflite，使用 SQLite 在设备本地持久化存储血糖历史记录
- 本地配置存储：shared_preferences，用于保存用户身体档案和计步基线等轻量配置
- 图表绘制：fl_chart，绘制血糖波动折线图
- 进度指示：percent_indicator，绘制步数目标环形进度
- 计步传感器：pedometer，读取设备内置加速度传感器的步数数据
- 权限管理：permission_handler，处理 Android 平台的活动识别权限申请
- 蓝牙通信：HC-05 蓝牙模块，负责与 STM32 硬件端的数据传输

## 三、应用架构

### 3.1 整体结构

应用采用分层架构组织代码，自底向上分为数据模型层、持久化层、状态管理层和界面层。

数据模型层定义了两个核心实体：GlucoseRecord（血糖记录）和 UserProfile（用户身体档案）。持久化层通过 DatabaseHelper（单例模式的 SQLite 封装）和 PreferencesHelper（SharedPreferences 封装）分别管理结构化数据和键值配置。状态管理层使用 Riverpod 的 Notifier/Provider 体系，将数据库读写、传感器监听、规则计算等逻辑封装为多个独立的 Provider，各 Provider 之间通过 ref.watch 建立依赖关系，实现数据变化时的自动联动。界面层消费 Provider 暴露的状态进行渲染，本身不包含业务逻辑。

### 3.2 页面结构

应用包含三个主要页面，通过底部导航栏切换，使用 IndexedStack 维持页面存活状态：

- 健康概览（Dashboard）：血糖数据可视化与蓝牙设备状态
- 运动与干预（Exercise）：运动数据展示与健康评估建议
- 设置（Settings）：用户身体档案管理与设备管理

## 四、功能模块详细说明

### 4.1 数据模型

#### 血糖记录模型（GlucoseRecord）

每条血糖记录包含以下字段：

- id：数据库自增主键
- timestamp：测量时间戳，精确到秒
- glucoseValue：血糖值，单位为 mmol/L
- timeTag：测量时间段标签
- baselineSteps：记录该条血糖数据时设备的累计步数基线，用于后续计算两次测量之间的实际步行量

系统定义了 7 个时间段标签，覆盖一天中血糖监测的典型场景：空腹、早餐后 2 小时、午餐前、午餐后 2 小时、晚餐前、晚餐后 2 小时、睡前。这些标签与血糖值共同构成健康评估引擎的输入参数。

#### 用户身体档案模型（UserProfile）

用户档案包含以下字段：

- age：年龄
- gender：性别（男/女）
- height：身高，单位 cm
- weight：体重，单位 kg
- hasDiabetes：是否已确诊糖尿病

模型内置了三项生理指标的计算方法：

1. BMI 计算：体重(kg) / 身高(m)²
2. 基础代谢率（BMR）计算：采用 Mifflin-St Jeor 公式，男性为 10×体重 + 6.25×身高 - 5×年龄 + 5，女性为 10×体重 + 6.25×身高 - 5×年龄 - 161
3. 步幅估算：根据性别和身高，男性取身高×0.415/100 米，女性取身高×0.413/100 米

### 4.2 本地持久化

#### SQLite 数据库（DatabaseHelper）

数据库采用单例模式管理，版本控制机制支持表结构的平滑升级，通过 onUpgrade 回调执行 ALTER TABLE 语句完成无损迁移。

提供的数据操作包括：插入单条记录、查询全部记录（按时间倒序）、按日期范围查询记录。

#### 配置存储（PreferencesHelper）

负责两类数据的本地持久化：

1. 用户身体档案：以 JSON 字符串形式序列化存储，应用启动时反序列化还原
2. 计步基线数据：存储当日日期字符串和对应的系统累计步数基线值，用于每日步数的差值计算

### 4.3 健康概览页（Dashboard）

该页面是用户打开应用后的默认页面，由三个卡片区域组成。

#### 蓝牙设备状态卡片

显示当前硬件设备的连接状态。设备通过 HC-05 蓝牙模块与手机通信，接收硬件端计算得到的血糖数据。

#### 血糖波动折线图

使用 fl_chart 绘制历史血糖数据的时序折线图。横轴为测量时间（显示时分），纵轴为血糖值（mmol/L）。图表支持以下特性：

- 数据点在状态管理层预先按时间戳升序排列，避免在界面渲染时反复排序消耗性能
- 纵轴范围根据数据动态调整，在最大值基础上预留 2-4 mmol/L 的余量空间，防止顶部数据点及其提示标签被截断
- 当记录数量较多时，图表区域支持横向滑动浏览，每个数据点分配约 60 逻辑像素宽度
- 折线下方绘制由主题色渐变至透明的填充区域，增强视觉层次

#### 今日步数进度卡片

以独立的 ConsumerWidget 实现，与主页面的重建周期隔离。该卡片监听计步传感器的实时数据流，以环形进度条的形式展示当日步数相对于 10000 步目标的完成率。

由于计步传感器的数据更新频率较高，将该卡片与主页面的折线图渲染解耦，避免步数变化时触发整个页面（包括图表）的无意义重建。

### 4.4 计步与每日步数计算

#### 传感器数据获取

应用通过 pedometer 插件监听 Android 设备的加速度传感器数据流。在启动监听前，先通过 permission_handler 向系统申请 ACTIVITY_RECOGNITION 权限。

#### 每日步数的差值计算

Android 系统的计步传感器返回的是自设备上次启动以来的累计总步数，而非每日步数。为了得到当天的实际步行量，应用采用本地基线差值法：

1. 每次接收到传感器事件时，读取当前日期字符串，与 SharedPreferences 中保存的基线日期比较
2. 若日期不同（跨日）或无历史基线（首次启动），将当前累计步数存为新的基线值，当日步数从 0 开始计
3. 若日期相同，当日步数 = 当前累计步数 - 基线值
4. 若当前累计步数小于基线值（设备重启导致计数器归零），将基线重置为 0，当日步数取当前累计步数

### 4.5 运动与干预页（Exercise）

该页面根据当日是否已有血糖记录，呈现两种不同的状态。

#### 无数据状态

当今日尚无血糖记录时，页面显示引导文案，提示用户先进行一次测量。

#### 有数据状态

当今日已有至少一条血糖记录时，页面依次展示三个模块：

1. 上次测量数据卡片：显示最近一次血糖值、对应的时间段标签和具体测量时间
2. 测后活动代谢卡片：显示自上次测量以来的有效步数和总消耗卡路里
3. 健康评估建议卡片：根据当前血糖值、时间段和活动量，由本地规则引擎生成的个性化建议

#### 卡路里消耗计算

卡路里消耗由 CalorieCalculator 纯函数计算，输入参数为用户身体档案、有效步数和经过时间，计算逻辑为：

1. 静息消耗：将 BMR（每日总基础代谢）按经过的小时数等比折算
2. 活动消耗：根据步数和步幅估算步行距离（公里），再按每公里每公斤体重消耗 1.036 大卡的系数折算
3. 总消耗 = 静息消耗 + 活动消耗

有效步数的计算方式为：当前系统累计步数减去上次记录血糖时保存的基线步数。若当前步数小于基线（设备重启），则直接取当前步数。

### 4.6 健康评估规则引擎（RecommendationEngine）

规则引擎是一个纯函数模块，接收四个输入：当前血糖值、测量时间段（空腹/餐前、餐后、睡前三类粗粒度）、累计活动卡路里、是否为糖尿病患者。引擎根据预设的阈值和条件组合，输出一条包含标题、详细建议文案和严重等级（危急/警告/提示/良好）的评估结果。

引擎的判定逻辑按优先级从高到低排列：

第一优先级为全局危险检测，无视时间段与活动量。血糖低于 3.9 mmol/L 时触发低血糖危急警报，提示立即补充碳水并禁止运动；血糖高于 13.9 mmol/L 时触发高血糖危急警报，提示多饮水并禁止剧烈运动。

第二优先级为空腹和餐前时段的综合评估。根据血糖是否达标（糖尿病患者目标上限 7.0，非糖尿病为 6.1）以及活动量是否充足，组合生成 6 种不同的建议方案，从"血糖完美达标"到"血糖明显偏高需即刻干预"。

第三优先级为餐后时段评估。根据餐后血糖是否达标（糖尿病患者上限 10.0，非糖尿病为 7.8）以及活动量的三个级别（不足 50 kcal、50-150 kcal、150 kcal 以上），生成 5 种建议方案，侧重于引导用户通过餐后运动降低血糖峰值。

第四优先级为睡前时段评估。检测睡前血糖是否偏低（低于 6.0 mmol/L 时提示夜间低血糖风险）或偏高，同时考虑活动消耗是否可能加剧夜间低血糖风险。

所有条件均未命中时，返回一条通用的正常状态提示。

### 4.7 设置页（Settings）

设置页面包含两个功能区域。

#### 用户身体档案表单

提供性别选择、年龄/身高/体重的数值输入框和糖尿病状态开关。表单初始加载时从 SharedPreferences 读取已保存的历史数据进行回填。用户点击保存后，数据同步写入本地存储，并通过 Riverpod 状态联动即时更新运动与干预页中依赖这些参数的计算结果。

#### 设备管理区域

提供蓝牙设备的扫描与绑定入口，用于管理与 HC-05 蓝牙模块的连接。

### 4.8 状态管理与数据流

应用使用 Riverpod 构建了一套响应式的数据流网络，各 Provider 之间的依赖关系如下：

- dashboardProvider（AsyncNotifier）：管理血糖记录列表，负责数据库的读写操作，新增记录后重新拉取并按时间排序后发布
- stepCountProvider（StreamProvider）：持续监听计步传感器数据流，经过每日基线差值转换后输出当日步数
- userProfileProvider（AsyncNotifier）：管理用户身体档案，从 SharedPreferences 加载并提供更新接口
- exerciseProvider（同步 Provider）：聚合上述三个 Provider 的数据，结合当前时间计算有效步数和卡路里消耗，生成 ExerciseState 数据包

当用户在首页记录一条血糖数据时，dashboardProvider 更新状态，exerciseProvider 因监听了 dashboardProvider 而自动重新计算，运动页的界面随之刷新。当用户在设置页修改身体参数时，userProfileProvider 更新状态，exerciseProvider 中的卡路里计算自动使用新参数重算。当计步传感器上报新的步数时，stepCountProvider 发出新值，首页的步数卡片和 exerciseProvider 同步更新。

### 4.9 界面设计

应用采用 Material Design 3 设计规范，整体风格为浅色调科技健康风。通过 AppTheme 统一定义了色彩体系：主色为青绿色（0xFF00CBA9），辅助色为湛蓝色（0xFF007AFF），警告色为红色（0xFFFF3B30）。卡片采用 24dp 大圆角和双层柔和阴影，按钮和输入框同样使用 16dp 圆角，整体视觉保持一致的圆润感。

## 五、数据隐私

应用的全部数据存储和计算均在用户设备本地完成。血糖记录存储于设备内部的 SQLite 数据库文件中，用户身体档案存储于 SharedPreferences 中，健康评估由本地规则引擎离线计算。应用不包含任何网络请求模块，不会将用户的健康数据上传至外部服务器。
